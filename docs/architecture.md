# CIRRADIO Architecture

Reference documentation for the CIRRADIO system design.

---

## 1. Key Hierarchy

CIRRADIO uses a three-level key hierarchy. All crypto operations run inside an
HSM; plaintext keys never exist in main RAM.

```
KEK  (Key Encryption Key)
 └─ stored in HSM, never exported
 └─ wraps/unwraps TEK and FHEK

TEK  (Traffic Encryption Key)
 └─ per-net session key
 └─ AES-256-GCM: encrypts all voice + data
 └─ rotated hourly via OTAR
 └─ distributed encrypted under new node's IK public key at net join

FHEK (Frequency-Hop Encryption Key)
 └─ separate key domain from TEK
 └─ used only by the AES-256-ECB hop sequencer
 └─ TEK compromise does not reveal the hop pattern

IK   (Identity Key)
 └─ per-device ECDSA P-384 keypair
 └─ provisioned at manufacturing
 └─ used only for net join authentication — never for traffic
```

Key storage uses the `IHsmEngine` interface. The `Pkcs11Hsm` implementation
dlopen-loads any PKCS#11 2.40-compliant HSM library at runtime; `SoftHsm`
(subclass) uses SoftHSM2 for development and CI.

## 2. Net Join Protocol

A new node joins the network using a challenge-response protocol authenticated
by ECDSA P-384 identity keys. Target join time: < 2 seconds.

```
New Node                          Existing Node
    │                                    │
    │── slot 19, rendezvous freq ───────▶│  (fixed frequency, not hopping)
    │                                    │
    │◀── nonce challenge ────────────────│
    │                                    │
    │── ECDSA P-384 sign(nonce, IK) ────▶│
    │                                    │  verify against trusted node list
    │                                    │
    │◀── {TEK, FHEK} encrypted under ───│  encrypt under new node's IK public key
    │    new node's IK public key        │
    │                                    │
    │── ACK ────────────────────────────▶│
    │                                    │
    │  begin hopping with net            │
```

If the identity key is not in the trusted node list, the challenge response is
silently dropped. Repeated failures trigger rate limiting.

## 3. FHSS Hop Sequencer

The hop sequence is generated by AES-256-ECB encryption of a structured plaintext.

**Plaintext format (16 bytes):**

```
 Byte  0:     slot number (0–19)
 Bytes 1–3:   padding (0x00)
 Bytes 4–7:   frame counter, little-endian uint32
 Bytes 8–14:  padding (0x00)
 Byte  15:    attempt number (0 = first try; incremented on blacklist hit)
```

**Channel selection:**

```
ciphertext = AES-256-ECB(FHEK, plaintext)
raw        = ciphertext[0:8] interpreted as little-endian uint64
channel    = raw mod 287
freq_kHz   = 225000 + channel * 1000   # 225–511 MHz, 1 MHz steps
```

**Blacklist:** If the selected frequency is in the blacklist (up to 20 jammed
frequencies in REG_BLACKLIST registers), increment `attempt` and recompute.
Up to 5 retries before accepting the least-bad channel.

**Timing:** The FHSS engine increments the frame counter on each GPS 1PPS pulse.
On GPS loss, the AXI Timer provides a holdover tick at the same rate.
REG_STATUS.STATUS_GPS_HOLDOVER_BIT is set during holdover.

## 4. TDMA Frame Structure

Each FHSS hop period contains one TDMA frame of 20 slots.

```
Slot 0:      Control / beacon (broadcast by net controller)
Slots 1–18:  Traffic (voice + data, assigned via REG_SLOT_BITMAP)
Slot 19:     Net join rendezvous (fixed frequency, not hopping)
```

Each node's slot assignments are written to REG_SLOT_BITMAP by the PS.
The TDMA MAC asserts TXNRX and the PA enable GPIO at the start of owned TX slots
and releases them at the slot boundary. Guard intervals prevent inter-slot
interference from transmit/receive switching transients.

## 5. AXI Register Map

The PS↔PL control plane is a 4 KB AXI4-Lite page at the base address
configured in the block design. All offsets are byte addresses.

### Write Registers (PS → PL)

| Offset  | Name               | Width | Reset      | Description                        |
|---------|--------------------|-------|------------|------------------------------------|
| 0x000   | REG_FHEK_0         | 32    | 0x00000000 | FHEK[31:0]                         |
| 0x004   | REG_FHEK_1         | 32    | 0x00000000 | FHEK[63:32]                        |
| 0x008   | REG_FHEK_2         | 32    | 0x00000000 | FHEK[95:64]                        |
| 0x00C   | REG_FHEK_3         | 32    | 0x00000000 | FHEK[127:96]                       |
| 0x010   | REG_FHEK_4         | 32    | 0x00000000 | FHEK[159:128]                      |
| 0x014   | REG_FHEK_5         | 32    | 0x00000000 | FHEK[191:160]                      |
| 0x018   | REG_FHEK_6         | 32    | 0x00000000 | FHEK[223:192]                      |
| 0x01C   | REG_FHEK_7         | 32    | 0x00000000 | FHEK[255:224]                      |
| 0x020   | REG_BLACKLIST[0]   | 32    | 0x00000000 | Jammed freq [0] in kHz             |
| …       | …                  | …     | …          | REG_BLACKLIST[1..19] at +4 each    |
| 0x06C   | REG_BLACKLIST[19]  | 32    | 0x00000000 | Jammed freq [19] in kHz            |
| 0x070   | REG_SLOT_BITMAP    | 32    | 0x00000000 | Bit N=1: this node owns slot N     |
| 0x074   | REG_TX_POWER       | 32    | 0x00000000 | TX power setpoint, dBm×100 signed  |

### Read Registers (PL → PS)

| Offset  | Name               | Width | Reset      | Description                        |
|---------|--------------------|-------|------------|------------------------------------|
| 0x080   | REG_STATUS         | 32    | 0x00000002 | Bit 0: hop lock; bit 1: GPS holdover|
| 0x084   | REG_RSSI           | 32    | 0xFFFF8300 | RSSI in dBm×100 (signed)           |
| 0x088   | REG_HOP_COUNTER    | 32    | 0x00000000 | [7:0]=slot, [31:8]=frame           |
| 0x08C   | REG_ERR_BASE[0]    | 32    | 0x00000000 | Per-slot error count, slot 0       |
| …       | …                  | …     | …          | REG_ERR_BASE[1..19] at +4 each     |
| 0x0D8   | REG_ERR_BASE[19]   | 32    | 0x00000000 | Per-slot error count, slot 19      |

The C++ mirror of this table is in `software/embedded/drivers/axi_regs.hpp`.
Run `python3 fpga/scripts/check_regmap.py` to verify they stay in sync.

## 6. Embedded HAL Composition

The same C++ comms stack binary runs on the Zynq ARM (real hardware) and on
any desktop (simulation). The only difference is which `IRadioHal` and
`IGpsHal` implementation is injected at construction time.

```
IRadioHal (interface)
  ├── SimRadioHal      — loopback shared-memory I/Q (desktop, all 97 tests)
  └── Zynq7045HAL     — composes three real drivers:
        ├── IioRadio  — AD9361 control + AXI DMA I/Q via libiio
        ├── AxiRegs   — UIO mmap of the 4 KB AXI4-Lite register page
        └── GpsHal    — gpsd JSON socket: position, time, PPS timestamp

IGpsHal (interface)
  ├── MockGpsHal      — returns fixed test coordinates
  └── GpsHal          — real gpsd socket implementation
```

`RadioNode` constructs the stack by accepting `IRadioHal&` and `IGpsHal&`
references. The `main.cpp` for embedded targets instantiates `Zynq7045HAL`
and passes it in; the test harness uses `SimRadioHal` and `MockGpsHal`.
